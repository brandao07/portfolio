# AGENTS.md

Guidance for future agents working on this Astro portfolio.

## Project Overview

This is a minimal personal portfolio built with Astro and TypeScript.

- `src/pages/` contains routes.
- `src/layouts/BaseLayout.astro` owns the shared document shell, top navigation, metadata, and footer slot.
- `src/components/` contains reusable Astro UI components.
- `src/data/` contains typed content/data used by pages and components.
- `src/styles/` contains global CSS split by concern.
- `src/lib/site-paths.ts` handles links that must respect `BASE_URL`.
- `public/assets/` contains static images and PDFs served directly by Astro.

Do not edit `dist/`; it is generated by `npm run build`.

## Common Commands

- `npm run dev` starts the local Astro dev server.
- `npm run check` runs Astro/TypeScript diagnostics.
- `npm run build` builds the static site and sitemap.

Before finishing code changes, run:

```bash
npm run check
npm run build
```

For content-only Markdown or documentation changes, a build is usually optional, but still useful if route/content config changed.

## Content And Data Rules

- Keep portfolio copy and typed entries in `src/data/`, not inline inside pages.
- Use existing data modules:
  - `site.ts` for metadata, navbar items, and social links.
  - `portfolio.ts` for homepage sections, experience, projects, skills, and education.
  - `research.ts` for research/publications.
  - `certificates.ts` for certificate entries.
  - `blog.ts` for blog archive helpers.
- If two features need the same list or value, extract a shared constant instead of duplicating it. Example: `yariLabsTechnologies` is shared by experience and certificates.
- Keep entries ordered in the data file when order matters. Do not sort in components unless there is a clear product reason.

## Components And Pages

- Pages should stay thin: import data, choose layout, compose components.
- Components should render one clear UI concept:
  - `SectionBlock` for section wrapper/copy.
  - `PageTeaser` for homepage CTAs to internal pages.
  - `ResourceLinks` for repeated link lists.
  - `CertificateList`, `PublicationList`, `ProjectGrid`, etc. for typed lists.
- Prefer extending existing components before creating near-duplicates.
- Keep interactive behavior minimal and local. Current shared navigation behavior lives in `BaseLayout.astro`.

## Routing And Links

- Use `toSitePath()` for internal links and asset paths rendered in components/layouts.
- Use `resolveLinkHref()` when a link may be internal, hash-based, or external.
- Internal page routes should include trailing slash paths in data/nav, e.g. `/research/`.
- Static assets in `public/assets/` are referenced as `/assets/...`.

## Styling Rules

- `global.css` only imports the CSS layers.
- `tokens.css` owns design tokens.
- `base.css` owns reset/base element styles.
- `layout.css` owns shell, topbar, nav, section layout, grids, footer, and route-level layout utilities.
- `components.css` owns shared card/button/chip/link primitives.
- `hero.css`, `projects.css`, and `pages.css` own more specific feature/page styling.
- Reuse existing tokens, card styles, buttons, chips, and spacing before adding new styles.
- Keep mobile behavior explicit with media queries. After navbar changes, check both desktop and mobile widths.
- Avoid one-off color values unless the design token set does not cover the case.

## Assets

- Put public static files under `public/assets/`.
- Certificate logos currently live in `public/assets/certificates/`.
- Prefer small optimized logos. WebP or SVG is ideal; JPEG is acceptable for small logos already in use.
- Do not introduce large unoptimized images without resizing/compressing them first.

## Feature Workflow

1. Inspect the relevant existing page, component, data module, and styles before editing.
2. Add or update typed data first.
3. Build or reuse a focused component.
4. Compose the component in the page.
5. Add scoped CSS in the most appropriate stylesheet.
6. Run `npm run check` and `npm run build`.
7. Review whether the feature changed project structure, conventions, commands, assets, routing, styling rules, or workflow expectations. If yes, update this `AGENTS.md` in the same change.
8. Review `git diff --stat` and `git status --short` before summarizing.

## Git Notes

- Keep commits focused.
- When a feature changes project conventions or structure, include the related `AGENTS.md` update in the same commit.
- Use Conventional Commit style when committing, for example:
  - `feat(portfolio): add certificates page`
  - `fix(nav): add mobile menu toggle`
  - `polish(nav): smooth mobile menu transitions`
- Do not push unless explicitly asked.
